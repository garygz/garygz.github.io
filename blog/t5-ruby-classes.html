Ruby Classes
What is it and how to use it
11/27/2014
Ruby classes are simply a way to create logical groups of functionality to increase code readability, maintainability and reliability. Class represents a type of an object which could be created to hold data and allow programmers to manipulate that data to mimic to perform useful tasks. For example, let's create a class represent a checking account: 
class CheckingAccount
end

So far this class does not do anything at all, so let's make a little more useful. A checking account should hold the data about you and the amount of dollars that you keep on your account. That means CheckingAccount should have instance variables(variables that represent the state of the object i.e. your account information). The best - more robust way to declare instance variables is by using attr_accessor keyword. Not only it defines clearly what the instance variables are for the class, but it also creates setter/getter methods for accessing and manipulating your data. Furthermore, we will add an initialize method (object constructor) so we can create a new object (i.e. open new checking account). 

  
class CheckingAccount
  attr_accessor :name, :address, :phone, :amount
  def initialize(name, address, phone, amount)
    @name = name
    @address = address
    @phone = phone
    @amount = amount
  end
end

#let's create two new accounts
davidsCheckingAccount = CheckingAccount.new("David","1 West St, Smallville, NY 11211", "617-111-1111", 100)
samsCheckingAccount = CheckingAccount.new("Sam","1 Liitle St, New York, NY 11211", "617-111-1111", 50)

puts "How much does David has on his account? $#{davidsCheckingAccount.amount}" #<= 100

# David deposits $100 so now he has $200
davidsCheckingAccount.amount = 200

puts "How much does David has on his account now? $#{davidsCheckingAccount.amount}" #<= 200

There are two things to note here:

      When declaring instance variable using attr_accessor each variable should start with a colon:name
      When using the instance variable anywhere withing the class @ sign should nbe used@name = name

As we see above David just deposited $100 to his account. All actions that can be performed on a class are methods. So let's write a method deposit which allow David to deposit any amount and then see the new balance: 


class CheckingAccount
  attr_accessor :name, :address, :phone, :amount
  def initialize(name, address, phone, amount)
    @name = name
    @address = address
    @phone = phone
    @amount = amount
  end

  def deposit(depositAmount)
    @amount = @amount + depositAmount
  end
end

#let's create two new accounts
davidsCheckingAccount = CheckingAccount.new("David","1 West St, Smallville, NY 11211", "617-111-1111", 100)
samsCheckingAccount = CheckingAccount.new("Sam","1 Liitle St, New York, NY 11211", "617-111-1111", 50)

puts "How much does David has on his account? $#{davidsCheckingAccount.amount}" #<= 100

# David deposits $100 so now he has $200
# Unlike in the previous example, we actually deposit the amount that David brings into the bank
# and let the class itself manipulate the data in order to set the correct balance.
# This is a concept of encapsulation - we let the class perform the correct action
# rather then deciding ourselves on to how to calculate the correct balance
davidsCheckingAccount.deposit (100)

puts "How much does David has on his account now? $#{davidsCheckingAccount.amount}" #<= 200

Good class design should be transparent, reusable, well encapsulated and loosely coupled. Transparent classes are classes that can be easily understood by other programmers. Their names and methods serve as basic documentation and are simple and logical. For example, the purpose and the expected functionality of a class named CheckingAccount is clear. However if we call it CAcct, the purpose of the class becomes unclear and thus not transparent.

Reusable classes are the ones that can be used in other contexts, for example CheckingAccount can be also easily used as savings account.

Well encapsulated classes do not expose and do not allow to modify it's instance variables to external programs and instead provide well defined interface(methods) which allow to access and manipulate data only within the allowable object state. For example, method deposit can check that the amount is not negative and will not allow to withdraw money instead of depositing.

Loosely coupled classes is a design pattern which promotes applications that are easi to maintain and modify. It means that a change in one class does not trigger changes in other classes. One such example would be to use Ruby HashMap constrictor arguments instead of a fixed list such as this:

Let's say we need to add an email instance variable as an optional field to CheckingAccount. 

class CheckingAccount
  attr_accessor :name, :address, :phone, :amount, :email
  def initialize(name, address, phone, amount, email)
    @name = name
    @address = address
    @phone = phone
    @amount = amount
    @email = email
  end

  def deposit(depositAmount)
    @amount = @amount + depositAmount
  end
end
Now this code will break meaning that our previous class was not loosely coupled:
davidsCheckingAccount = CheckingAccount.new("David","1 West St, Smallville, NY 11211", "617-111-1111", 100) However, if we re-write the constructor to use a hash map: 

def initialize(args)
    @name = args.fetch(:name)
    @address = args.fetch(:address)
    @phone = args.fetch(:phone)
    @amount = args.fetch(:amount)
    @email = args.fetch(:email, "N/A")
end
and change the instance creation method call to use a hash map as well:

To summarize: classes are just a way to group variables and methods into small logical groups which are easy to maintain, modify, re-use and understand.
